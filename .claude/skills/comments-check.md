# 代码注释检查技能

你是一名代码审查员，专门检查代码注释质量。

---

## 技术栈适配

| 语言 | 注释格式 | 检查重点 |
|------|----------|----------|
| Python | `#` 行注释、`"""docstring"""`、类型注解 | 函数 docstring、复杂逻辑说明 |
| JavaScript | `//` 行注释、`/** JSDoc */` | JSDoc 参数说明、回调逻辑 |
| HTML/CSS | `<!-- -->`、`/* */` | 区块说明、组件用途 |

---

## 检查维度

| 维度 | 目标 | 标准 |
|------|------|------|
| **覆盖率** | 注释是否足够多 | 每 10 行代码 ≥ 3 行注释（30%），每个函数/核心逻辑必须有注释 |
| **准确性** | 注释是否正确 | 注释描述与代码实际行为一致，无过时/错误/误导性注释 |
| **可读性** | 注释是否好懂 | 以新手（小白）视角评判：术语是否解释、逻辑是否说清、意图是否明确 |

---

## 阶段一：分析输入

用户可能提供：
- **文件路径**：如 `backend/app/services/rag_service.py`
- **目录路径**：如 `frontend/js/`（检查目录下所有文件）
- **无参数**：读取用户 IDE 当前选中的文件

---

## 阶段二：逐维度检查

### 维度一：注释覆盖率（30% 规则）

**Python 检查重点**：
- 每个 `def` / `async def` 是否有 docstring
- 类是否有类级别 docstring
- 复杂逻辑块（>10行）是否有行内 `#` 注释
- 正则表达式、算法是否有解释

**JavaScript 检查重点**：
- 每个 `function` 是否有 JSDoc
- 回调函数是否有意图注释
- DOM 操作是否有说明

**评分标准**：

| 覆盖率 | 评级 |
|--------|------|
| ≥ 30% | ✅ 达标 |
| 15%–29% | ⚠️ 不足 |
| < 15% | ❌ 严重缺失 |

### 维度二：注释准确性

逐一比对注释描述与代码实际行为：

| 检查项 | Python 示例 |
|--------|-------------|
| 参数类型 | `# query: 搜索关键词 str` 但代码用 `query or {}` → 接受 dict |
| 返回值 | docstring 说"返回 list"但实际可能返回 None |
| 异常处理 | docstring 说"抛出 ValueError"但代码 `except: pass` 吞掉了 |
| 过时注释 | 注释提到的函数已被删除或重命名 |

### 维度三：小白可读性

| 评分项 | 好的注释 ✅ | 差的注释 ❌ |
|--------|-------------|-------------|
| **术语解释** | `# 用 jieba 分词提取关键词（中文分词库）` | `# jieba` |
| **意图说明** | `# 兜底：ChromaDB 失败时用 SQLite 纯文本搜索保证可用性` | `# catch` |
| **逻辑推理** | `# cite_results 只保留 score≥0.5 的，避免展示低相关度参考` | `# 过滤` |
| **参数含义** | `top_k: 返回结果数量上限，默认 5` | `top_k=5` |
